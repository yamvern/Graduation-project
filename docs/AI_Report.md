## تقرير المكوّنات الذكية (AI Report)

### التحقق بالوجه (DeepFace – FaceService)
- **الموقع في المشروع**: `ai/Biometric/face_service.py` ويُستخدم في `api/routers/biometric_router.py`, `api/routers/face_router.py`, `api/services/verification_steps_service.py`
- **الغرض / الوظيفة**: مطابقة وجهين عبر DeepFace (افتراضيًا Facenet + cosine). يحوّل البايتات إلى صور OpenCV ثم يستدعي `DeepFace.verify` ليعيد المسافة والتشابه والقرار.
- **الاعتمادات**: `deepface`, `tensorflow/keras` (أو `tf-keras`), `opencv-python`, `numpy`.
- **المدخلات / المخرجات**: مدخلات: صورتان (بايتات). مخرجات: `{match, distance, similarity, similarity_percent, threshold}`.
- **المخاطر / المشكلات**: تبعيات ثقيلة؛ فشل إذا لم تُحمَّل النماذج أو كانت البيئة بلا GPU؛ لا يوجد تحقّق مسبق لحجم/نوع الصورة؛ زمن استجابة مرتفع على CPU.
- **التوصيات**: إضافة فحص حجم/نوع؛ تمهيد/كاش للنماذج؛ توثيق الاعتمادات؛ توفير نموذج أخف أو طابور للطلبات.

### راوتر التحقق البيومتري (API)
- **الموقع**: `api/routers/biometric_router.py`
- **الغرض / الوظيفة**: مسار `POST /api/v1/biometric/verify` يقوم بـ liveness بسيط ثم FaceService (DeepFace)، ويسجّل النتيجة في `biometric_audit_log`.
- **الاعتمادات**: FaceService + DeepFace stack، `opencv-python`, `numpy`، اتصال قاعدة بيانات.
- **المدخلات / المخرجات**: form-data: `selfie_image`, `document_image`, `user_id`, `document_id`. المخرجات: نتيجة الحيوية + نتيجة المطابقة (distance, similarity, threshold).
- **المخاطر / المشكلات**: liveness بدائي (تباين/حجم فقط) → قابل للتحايل؛ يعتمد على توفر DeepFace؛ لا حدود حجم/نوع للملفات.
- **التوصيات**: تحسين liveness (anti-spoofing)؛ فرض قيود حجم/نوع؛ جعل العتبة قابلة للضبط؛ معالجة graceful عند تعذّر DeepFace.

### راوتر التحقق بالوجه (قديم)
- **الموقع**: `api/routers/face_router.py`
- **الغرض / الوظيفة**: مسار `POST /face/verify` يطابق صورتين باستخدام FaceService ويسجّل في سجل التدقيق.
- **الاعتمادات**: FaceService (DeepFace), audit_log_service.
- **المدخلات / المخرجات**: ملفان `photo1`, `photo2`؛ نفس ناتج FaceService.
- **المخاطر / المشكلات**: بلا حماية Auth؛ لا liveness؛ نفس مخاطر التبعيات.
- **التوصيات**: حماية بالتصريح؛ توحيد المنطق مع الراوتر البيومتري وإعادة استخدام فحص الحيوية.

### خدمة خطوات التحقق من الوثائق
- **الموقع**: `api/services/verification_steps_service.py`
- **الغرض / الوظيفة**: سلسلة خطوات للتحقق من الوثائق (جودة صورة، تصحيح، تخطيط، استخراج وجه، مطابقة وجه، OCR، فحص شعار/ختم، بلوكشين/IPFS).
- **الاعتمادات**: `opencv`, `numpy`, `yaml`, `subprocess`, FaceService, IPFSService, Hyperledger Fabric helpers, LogoAndStamp، OCR.
- **المدخلات / المخرجات**: مسارات ملفات؛ تعيد تقارير لكل خطوة.
- **المخاطر / المشكلات**: يستدعي ملفات محذوفة `ai/card_verification` (ستفشل)؛ استدعاءات subprocess بلا مهلات؛ مسارات ثابتة للテンبلت؛ تحقّق ضعيف.
- **التوصيات**: فحص وجود الملفات قبل الاستدعاء؛ إضافة مهلات؛ تفعيل/تعطيل الخطوات حسب توفر الموارد؛ توثيق المسارات.

### خط أنابيب فحص الشعار/الختم (Logo & Stamp)
- **الموقع**: `ai/LogoAndStamp/` (main_unified.py وبقية المجلدات)
- **الغرض / الوظيفة**: كشف تزوير الشعار/الختم على الوثائق باستخدام نماذج Torch؛ يُستدعى من `ml_verify`.
- **الاعتمادات**: `torch`, `torchvision`, `opencv`, `scikit-image`, `Pillow`, `PyYAML`, `pdf2image`, `fpdf2`.
- **المدخلات / المخرجات**: ملف صورة وثيقة → تقرير JSON موحّد (قرارات logo/stamp وأكواد overlay).
- **المخاطر / المشكلات**: تبعيات ثقيلة؛ احتمال نقص أوزان النماذج؛ يعتمد على GPU لأداء جيد؛ استدعاء subprocess قد يفشل بصمت.
- **التوصيات**: توثيق موقع الأوزان؛ فحص جاهزية النماذج؛ إضافة مهلات واسترجاع أخطاء واضحة؛ خيار تخفيف الحجم/النماذج.

### خدمة OCR (Google Vision)
- **الموقع**: `ai/ocr/vision_service_ocr.py`
- **الغرض / الوظيفة**: OCR للصور وPDF عبر Google Vision API؛ يُستخدم في خطوات التحقق.
- **الاعتمادات**: `requests`, `PyMuPDF (fitz)`, مفتاح Google Vision (`GOOGLE_VISION_API_KEY`).
- **المدخلات / المخرجات**: بايتات صورة/ PDF → نص + raw عند غياب نص.
- **المخاطر / المشكلات**: إرسال بيانات حساسة إلى خدمة سحابية (خصوصية)؛ يتطلب مفتاح؛ لا يوجد retry/backoff.
- **التوصيات**: حد لحجم/صفحات؛ مهلات وإعادة محاولات؛ تنبيه الخصوصية؛ توفير OCR محلي (Tesseract) كخيار.

### عميل IPFS
- **الموقع**: `ledger/ipfs_service.py` (نسخة مشابهة في `infrastructure/ipfs_service.py`)
- **الغرض / الوظيفة**: رفع/جلب ملفات إلى/من IPFS Kubo عبر HTTP.
- **الاعتمادات**: `requests`, `json`.
- **المدخلات / المخرجات**: بايتات/مسار ↔ CID؛ جلب bytes.
- **المخاطر / المشكلات**: يفترض تشغيل IPFS على المنفذ المضبوط؛ لا retries/مهلات؛ لا حدود حجم؛ ازدواجية client.
- **التوصيات**: توحيد عميل واحد؛ إضافة مهلات/إعادة محاولة؛ فرض حجم أقصى للملفات.

### مسار البلوكشين + IPFS (ميتاداتا)
- **الموقع**: `api/routers/blockchain_router.py`, `api/services/multichain_service.py`, `api/services/hash_service.py`
- **الغرض / الوظيفة**: حساب SHA-256، التحقق من التكرار (DB)، رفع الملف إلى IPFS، نشر metadata (HEX) في stream `documents` على MultiChain، منع التكرار.
- **الاعتمادات**: `hashlib`, subprocess مع `multichain-cli`, قاعدة بيانات (`document_hashes`).
- **المدخلات / المخرجات**: رفع ملف → `{document_id, cid, hash, metadata}`؛ قراءة stream → JSON مفكوك.
- **المخاطر / المشكلات**: يتطلب إعداد multichain-cli وstream؛ لا تحقق حجم/نوع قبل الرفع؛ معالجة أخطاء CLI محدودة.
- **التوصيات**: فحص الملفات قبل الرفع؛ مهلات وإعادة محاولات للـ CLI؛ توثيق إعداد السلسلة/stream.

### سكربتات تدريب/بيانات (تواقيع)
- **الموقع**: ملفات عديدة في الجذر `ai/` مثل `train_siamese.py`, `finetune_resnet18_custom.py`, `evaluate_siamese.py`, إلخ.
- **الغرض / الوظيفة**: توليد بيانات وتدريب/تقييم نماذج توقيع/Siamese.
- **الاعتمادات**: `torch`, `torchvision`, `opencv`, `numpy`, (غالبًا matplotlib).
- **المدخلات / المخرجات**: مجموعات صور تدريب → أوزان/تقارير.
- **المخاطر / المشكلات**: غير مدمجة في الـ API؛ تتطلب GPU غالبًا؛ مسارات بيانات/أوزان غير موثقة.
- **التوصيات**: تشغيلها في بيئة تدريب منفصلة؛ توثيق مسارات البيانات والأوزان؛ عدم تشغيلها في بيئة الإنتاج.

### التحقق من البطاقة (Card Verification)
- **الموقع**: يُشار إليه في `api/services/verification_steps_service.py` (مسارات `ai/card_verification/...`) لكن المجلد محذوف.
- **الغرض / الوظيفة**: تصحيح بطاقة، التحقق من التخطيط، استخراج وجه من الهوية اليمنية.
- **الاعتمادات**: OpenCV, YAML, سكربتات خاصة.
- **المدخلات / المخرجات**: صورة بطاقة → صورة مصححة/وجه.
- **المخاطر / المشكلات**: الملفات مفقودة حاليًا؛ أي استدعاء سيكسر الخطوة.
- **التوصيات**: إما استعادة الملفات أو تعطيل هذه الخطوات مع فحص وجود المسارات.

### راوتر وجه قديم (Biometric_api)
- **الموقع**: `api/Biometric_api/face_router.py`
- **الغرض / الوظيفة**: مسار آخر لمطابقة الوجه باستخدام `ai.face_service` (نفس FaceService).
- **الاعتمادات**: DeepFace stack.
- **المدخلات / المخرجات**: صورتان؛ نتيجة DeepFace.
- **المخاطر / المشكلات**: تكرار مع `/api/v1/biometric/verify` و`/face/verify`; قد يسبب ارتباكًا؛ تحكم Auth غير واضح.
- **التوصيات**: توحيد نقاط الدخول؛ فرض Auth؛ توثيق مسار واحد معتمد.

### OCR / Logo / Stamp + Blockchain (خطوة مجمّعة)
- **الموقع**: داخل `api/services/verification_steps_service.py` كمنسق رئيسي.
- **الغرض / الوظيفة**: دمج خطوات متعددة (OCR، مطابقة الوجه، ML للشعار/الختم، رفع IPFS، Fabric/Blockchain).
- **الاعتمادات**: جميع المكونات أعلاه.
- **المدخلات / المخرجات**: مسارات ملفات → تقارير متعددة + آثار جانبية (IPFS، Fabric، Blockchain).
- **المخاطر / المشكلات**: كثرة subprocess؛ اعتماد على موارد مفقودة؛ تأخير زمني؛ أخطاء متتالية.
- **التوصيات**: تمكين/تعطيل الخطوات حسب التوفر؛ إضافة مهلات؛ مركزية معالجة الأخطاء؛ التحقق المسبق من الموارد.

### اعتماد Google Vision / المفتاح
- **الموقع**: `.env` يحتوي `GOOGLE_VISION_API_KEY` ويُستخدم في `ai/ocr/vision_service_ocr.py`
- **الغرض / الوظيفة**: تزويد مفتاح لـ Google Vision OCR.
- **المخاطر / المشكلات**: غياب المفتاح يرفع استثناء عند الاستيراد؛ خصوصية البيانات المرسلة.
- **التوصيات**: تحميل كسول للخدمة؛ توفير بديل محلي؛ تنبيه خصوصية واضح.
